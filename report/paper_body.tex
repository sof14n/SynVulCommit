\section{Introduction}
Software vulnerabilities remain a high-impact source of security failures, and source-code learning methods are increasingly used to prioritize or classify vulnerable code. Their effectiveness depends on the quality, coverage, and labeling discipline of training data. Real repositories provide valuable examples, but vulnerability-fixing commits are sparse, noisy, and unevenly distributed across weakness types. VUDENC addresses this problem for Python by deriving vulnerable and fixed code from natural vulnerability-fixing commits \cite{vudenc}. However, a mined corpus cannot guarantee uniform coverage of a chosen CWE, framework, program structure, or data-flow pattern.

This work investigates whether an LLM can generate useful, commit-like Python vulnerability data while maintaining a strong validation boundary. SynVulCommit produces paired vulnerable and fixed files rather than isolated snippets. The objective is not to claim that LLM-generated data replaces real commits. Instead, the system creates a controlled corpus that supports an explicit experiment: compare models trained on real data, synthetic data, and their combination using a shared real-data test set.

The contributions are:
\begin{enumerate}
  \item A generation pipeline for seven Python CWE categories with deterministic quota-based coverage over application type, flow pattern, difficulty, and program structure.
  \item A layered acceptance policy combining CWE-specific structural checks, Bandit, Semgrep, a blinded LLM reviewer, and near-duplicate rejection.
  \item A clean VUDENC-style training export and a separate sanitized audit sidecar.
  \item A controlled A/B/C evaluation using eight model variants from the earlier lab work, including Word2Vec, CodeBERT, and GraphCodeBERT representations.
\end{enumerate}

\section{Background and Related Work}
Deep learning for vulnerability detection has progressed from code-gadget and representation-learning approaches, such as VulDeePecker \cite{vuldeepecker} and early deep representation learning \cite{russell2018}, to systematic vulnerability representations \cite{sysevr} and graph-based program semantics \cite{devign}. Dataset construction remains central: Big-Vul demonstrates the value of commit-level vulnerability data with code changes and CVE summaries \cite{bigvul}, while VUDENC specifically provides a Python corpus based on natural vulnerable/fixed commits \cite{vudenc}. Reviews of the area caution that reported performance can depend strongly on dataset construction and evaluation setup \cite{chakraborty2022}.

Pre-trained code models offer richer representations. CodeBERT jointly pre-trains on programming and natural-language data \cite{codebert}; GraphCodeBERT adds data-flow structure \cite{graphcodebert}; and VulBERTa targets vulnerability detection through simplified source-code pre-training \cite{vulberta}. Lab 3 evaluated Word2Vec, CodeBERT, and GraphCodeBERT model variants and found that transformer embeddings were not consistently more stable than Word2Vec baselines. For this project we therefore include both families rather than treating transformer results as out-of-scope.

LLMs can both assist development and introduce risks. Pearce et al. found insecure code patterns in generated Copilot suggestions \cite{pearce2022}. Conversely, LLM-based augmentation has been studied for long-tail vulnerability detection \cite{deng2024}. These findings motivate a conservative design: LLMs propose candidates, but deterministic analysis and independent review decide whether candidates enter the corpus.

\section{SynVulCommit Methodology}
\subsection{Generation Scope and Quota Planning}
The pipeline targets SQL injection (CWE-89), command injection (CWE-78), path traversal (CWE-22), open redirect (CWE-601), remote code execution (CWE-94), cross-site scripting (CWE-79), and cross-site request forgery (CWE-352). For each requested sample, a deterministic planner selects a compatible tuple of application type, data-flow pattern, difficulty, and structure. The planner favors the least-represented full tuple, then the least-represented marginal values, with a stable SHA-256 tie-breaker. A rejected candidate retries the same planned slot; it is not replaced by a random context.

The LLM returns a strict JSON object containing a commit message, relative filename, vulnerable code, fixed code, and exact changed lines. The prompt requires one requested CWE, a behavior-preserving fix, and no unrelated security-sensitive flow. Provider configuration is externalized through environment variables; keys, endpoints, raw replies, and authorization data are never stored in training exports.

\subsection{Validation and Review}
Figure~\ref{fig:pipeline} summarizes the acceptance path. Structural checks verify the intended source, sink, framework context, and fix strategy. Bandit and Semgrep then analyze vulnerable and fixed files. The requested rule must appear before the fix where configured, and no configured SynVulCommit CWE rule may remain after the fix. This cross-CWE post-fix rule prevents, for example, accepting an XSS fix that introduces SQL injection.

\label{fig:pipeline}

After deterministic validation, a second-stage reviewer receives the requested CWE/context and the vulnerable/fixed code, but not analyzer output or provider metadata. It returns a strict structured verdict. A candidate is accepted only when the review status is complete and the verdict is \texttt{pass}. This is a quality gate, not a replacement for static analysis.

\subsection{Diversity, Export, and Verification}
The diversity index rejects candidates in policy order: exact normalized vulnerable/fixed pair hashes, normalized AST-pair fingerprints, and same-CWE token-shingle similarity at Jaccard similarity at least 0.90. AST normalization removes identifiers, names, aliases, attributes, and literal-specific variation, preventing superficial renaming from creating a new sample.

Accepted records are exported to \texttt{plain\_<mode>} VUDENC-style files. These files retain only the commit message, vulnerable and fixed sources, diff, filename, changed-line counts, \texttt{badparts}, and \texttt{goodparts}. The separate \texttt{metadata.jsonl} sidecar contains sanitized provenance and validation summaries. A strict dataset verifier checks the accepted records, rejected categories, duplicate policy, reviewer state, and exact linkage between JSONL records and exported commits.

\subsection{Window-Balanced Generation Profile}
The first compact generation profile produced valid commit pairs, but many files were shorter than one 200-token model window. This caused most synthetic windows to overlap the vulnerable region. To align generation with the downstream VUDENC-style window task, the final corpus uses a \texttt{window\_balanced} profile. It requires realistic 420--900 token Python modules, clean context before and after the vulnerable region, localized vulnerable fragments, at least one vulnerable and one clean 200-token window, and fixed code that preserves most of the original file length. These checks are applied only to the window-balanced profile so the original compact generator remains available for diagnostics.

\section{Experimental Design}
\subsection{Datasets and Splits}
The final verified generation output contains 1,205 accepted window-balanced pairs. Figure~\ref{fig:coverage} shows the per-CWE counts. SQL injection is the limiting class with 149 accepted pairs, so the experiment freezes exactly 149 synthetic commits per CWE, for 1,043 synthetic commits total. The selection is deterministic and stratified by application type, flow pattern, difficulty, and structure.

\begin{figure}[t]
\centering
\includegraphics[width=\linewidth]{corpus_coverage.pdf}
\Description{Bar chart showing accepted pairs per CWE. SQL has 149 accepted pairs and XSRF has 200, with other CWEs between those values.}
\caption{Verified accepted pairs in the final window-balanced corpus. The dashed line marks the 149-pair-per-CWE frozen subset used for evaluation.}
\label{fig:coverage}
\end{figure}

Real VUDENC data is normalized into the same commit-level representation; the legacy path-disclosure mode is mapped to directory traversal. The real-data cap is $\min(149, \text{available commits})$ per CWE. The selected real commit counts are 149 SQL, 106 command injection, 140 directory traversal, 93 open redirect, 54 remote code execution, 69 XSS, and 141 XSRF. Commits, not windows, are deterministically split approximately 70/15/15 into train, validation, and test. Consequently, every window from a commit stays in one split.

Windows contain up to 200 lexical code tokens. A vulnerable-source window overlapping a \texttt{badpart} is labeled vulnerable; other source windows are labeled clean. For computational tractability, each file retains at most four positive and four negative windows. Figure~\ref{fig:labels} shows that the final synthetic split is no longer positive-skewed: synthetic training contains 2,908 vulnerable and 2,898 clean windows, while synthetic validation contains 612 vulnerable and 616 clean windows.

\subsection{A/B/C Conditions}
Condition A trains and validates on real VUDENC data. Condition B trains and validates on the synthetic subset. Condition C trains on the union of real and synthetic training windows and validates on real VUDENC. Every condition is evaluated on the identical held-out real VUDENC test split. Synthetic test windows are prepared for audit symmetry but are not used for the main reported evaluation.

\subsection{Model Families}
The experiment reuses the model families from Lab 3. The Word2Vec models use a 300-dimensional Python Word2Vec representation rebuilt from the existing external Python corpus when the saved legacy vector sidecar was unavailable. LSTM and CNN consume 200-token sequences with fixed Word2Vec embeddings. MLP and Random Forest use mean-pooled Word2Vec vectors. The Word2Vec models are:
\begin{itemize}
  \item LSTM: 100 units and 0.2 dropout;
  \item MLP: 256 and 128 ReLU hidden units with Adam;
  \item Random Forest: 300 trees and balanced class weights;
  \item CNN: 128 one-dimensional filters, kernel size 5, global max pooling, dense 128, and 0.2 dropout.
\end{itemize}

The transformer variants mirror the Lab 3 setup by using CodeBERT and GraphCodeBERT as frozen embedding extractors. For each 200-token window, the transformer hidden states are cached and used by either the same LSTM head or the same CNN head. The evaluated variants combine LSTM and CNN heads with CodeBERT and GraphCodeBERT embeddings. Neural models use Adam, batch size 128, up to 100 epochs, and early stopping based on validation F1. Python, NumPy, scikit-learn, and TensorFlow seeds are fixed.

The strongest overall result is MLP with mixed real-plus-synthetic training, with a mean macro-F1 of 0.7161. Random Forest trained on real data alone is second at 0.6984. Mixed training improves MLP and LSTM over their real-only Word2Vec baselines, but it does not improve Random Forest or CNN. Synthetic-only training remains the weakest condition for every model family except LSTM+CodeBERT, where all three conditions are weak and close together.

The transformer variants do not outperform the best Word2Vec models. The best transformer result is CNN+GraphCodeBERT with mixed training at 0.6232 macro-F1. This is competitive with the Word2Vec LSTM and CNN results, but below MLP and Random Forest. In this setup, frozen transformer embeddings appear to help less than a simpler pooled Word2Vec MLP trained with mixed data.

\section{Discussion and Limitations}\label{sec:discussion}
The result answers the research question conservatively. SynVulCommit can produce a validated, diverse, window-balanced synthetic corpus, and that corpus can improve some real-data models when used as augmentation. However, synthetic-only training still transfers poorly to held-out real VUDENC windows. The main remaining limitation is therefore not label imbalance; the window-balanced generation profile addressed that. The remaining limitation is distribution shift.

Real VUDENC commits come from actual projects, with inconsistent developer style, project-specific helper functions, framework conventions, partial files, and surrounding maintenance logic. Synthetic examples are cleaner and more regular, even when quota planning and duplicate rejection increase diversity. In addition, the acceptance policy deliberately favors samples that deterministic rules and the reviewer can verify. This improves label reliability, but it narrows the accepted distribution toward obvious and analyzable cases. A model trained only on synthetic data may therefore learn generated-code regularities rather than robust real-project vulnerability signals.

The transformer results reinforce this interpretation. CodeBERT and GraphCodeBERT embeddings do not automatically overcome the synthetic-to-real gap. Because the transformer encoders are used as frozen feature extractors rather than fine-tuned end-to-end, they provide richer token representations but still depend on the training distribution presented to the classifier head. The best overall result is a simpler MLP with mixed real and synthetic data, suggesting that data-condition design matters more here than representation complexity alone.

Several threats to validity remain. Static analysis and review improve confidence but do not prove exploitability or production realism. The reviewer is an LLM and can share blind spots with the generator, even when configured independently. The corpus is Python-only and covers seven CWEs, so the results should not be generalized to all languages or weakness types. The real VUDENC test split is small for remote code execution and XSS because VUDENC has fewer commits in these modes. Finally, this study uses one fixed seed for the final experiment; repeated multi-seed training would be needed to estimate variance.

\section{Conclusion}
SynVulCommit operationalizes controlled synthetic vulnerability-commit generation with quota-based context coverage, layered validation, reviewer gating, near-duplicate control, sanitized export, strict dataset verification, and window-balance checks aligned with the downstream VUDENC representation. The final verified corpus contains 1,205 accepted Python vulnerability-fix pairs, from which a balanced 1,043-pair subset was used for A/B/C model evaluation.

The experiments show that synthetic data alone is not a replacement for real VUDENC data in this task. However, synthetic data is useful as augmentation for some models: the best overall result is MLP with mixed real-plus-synthetic training, achieving a mean macro-F1 of 0.7161. The transformer variants from Lab 3 were also evaluated, but the best transformer model, CNN+GraphCodeBERT with mixed training, reached 0.6232 and did not outperform the strongest Word2Vec models. The practical conclusion is that validated synthetic commits can support controlled data-generation research and improve selected model families, but generation realism and downstream representation must be treated as part of the same system.
